## 一、项目概述

### 1.1 我们要解决什么问题

景区游览中，游客获取讲解的方式仍然停留在"扫码听语音"的时代——体验割裂、内容死板、千人一面。

**我们的方案**：基于Rokid AI眼镜，打造"看到即学到"的智能景区讲解系统。游客戴上眼镜，看向哪里，讲解就跟到哪里——像有一个博学又风趣的私人导游随时在身边。

### 1.2 核心价值

| 维度 | 传统方式 | 我们的方案 |
|------|---------|-----------|
| 操作方式 | 扫码/按键/举手机 | 抬头即见，无感触发 |
| 讲解内容 | 念百度百科 | AI讲故事，像真人导游 |
| 个性化 | 所有人听一样的 | 千人千面，按兴趣调整 |
| 语言 | 仅中文/需另请 | 一键多语言切换 |
| 无障碍 | 基本没有 | 实时字幕，听障可用 |

## 二、痛点分析

### 2.1 景区讲解的五大痛点

**痛点一：导游贵且难约**
- 旺季导游200-500元/次，热门景点需提前数天预约
- 淡季导游质量参差不齐，体验不可控

**痛点二：自助讲解体验差**
- 扫码听语音：需要掏手机、打开小程序、对准二维码，打断游览节奏
- 讲解器：笨重、按键复杂、老年人不会用

**痛点三：内容死板，像念说明书**
- 现有AI讲解 = 把百度百科念给你听
- 罗列数据（"高35.05米，建于1420年"），没有故事感
- 所有游客听到完全一样的内容

**痛点四：信息密度不可控**
- 想快速走走听听，讲解太长拖节奏
- 想深入了解，又没有更多内容
- 不会根据游客的停留行为自动调节

**痛点五：语言障碍与无障碍缺失**
- 外国游客听不了中文讲解，多语言导游稀缺
- 听障游客几乎无法获取任何讲解信息

### 2.2 现有AI讲解产品的问题

| 产品 | 形式 | 核心问题 |
|------|------|---------|
| 百度地图景区讲解 | 语音播报 | 念百科，无互动，内容死板 |
| 高德AR导览 | AR导航+语音 | 导航为主，讲解为辅，内容浅 |
| 各景区自有小程序 | 扫码听讲解 | 体验割裂，质量参差 |

**我们的机会**：所有现有产品都没有解决"让AI讲解听起来像真人导游"这个问题。
这正是AI大模型的能力所在——也是我们的核心切入点。

## 三、产品方案

### 3.1 产品形态

**硬件载体**：Rokid AI眼镜
**软件系统**：AI景区讲解Agent
**交互方式**：视觉触发 + 语音对话 + AR信息显示

### 3.2 核心功能

#### 功能一：所见即所讲
- 眼镜摄像头识别当前视野中的景点
- 自动触发该景点的讲解内容
- 无需任何操作，抬头就开始讲

#### 功能二：AI讲故事，不念说明书
- 基于Gemini大模型实时生成讲解内容
- 有故事线、有情绪、有互动，像真人导游在聊天
- 三级讲解深度，根据游客停留时间自动调节

#### 功能三：语音问答互动
- 游客随时提问："这个雕塑是谁？""这棵树多少年了？"
- AI结合视觉信息+知识库实时回答
- 像跟朋友聊天一样自然

#### 功能四：千人千面
- 首次使用选择风格：故事版/专业版/轻松版
- 根据兴趣偏好调整讲解角度（历史迷/建筑控/随便逛逛）
- 记住游览历史，不重复、能推荐

#### 功能五：多语言无障碍
- 一键切换讲解语言（中/英/日/韩）
- 听障模式：语音讲解→AR实时字幕
- 适老化：语速可调、大字显示

### 3.3 用户体验流程

```
游客戴上眼镜，进入景区
        │
        ▼
眼镜自动识别景点，开始讲解（基础层·10秒）
        │
        ├── 继续走 → 切换下一景点
        │
        ├── 停留 >3秒 → 展开标准讲解（30秒）
        │
        ├── 停留 >10秒 → 深度讲解（2分钟+）
        │
        ├── 游客提问 → AI实时回答
        │
        └── 游客说"换个语言" → 切换语言
```

### 3.4 应用场景

**场景一：故宫游览**
游客走向太和殿，眼镜自动识别→AR浮现"太和殿"标签→开始讲解：
> "你现在看到的这个大家伙叫太和殿。它大到什么程度呢？你把屋顶掀了，能塞进去一个足球场。六百多年前它就站在这儿了，比我们在座所有人的年龄加起来都大。"

游客抬头看屋脊→触发脊兽专题讲解：
> "屋脊上那一排小兽，不是装饰品。古人觉得屋顶最容易着火，就放了一排'神兽'当保镖。领头的是骑凤仙人，后面跟着龙、凤、狮子……一共十个，全国只有这一处能凑齐十个，所以太和殿的规格是最高的。"

游客问"为什么是十个？"→AI回答：
> "好问题！脊兽的数量代表建筑等级，皇帝住的地方最多也就九个。太和殿是举行大典的地方，比皇帝睡觉的地方还高级，所以破例给了十个。这待遇就跟现在国务院和省政府的区别差不多。"

**场景二：自然景区导览**
黄山游览，识别迎客松→
> "这就是迎客松，黄山的招牌。它站在这儿至少八百年了，根扎在花岗岩缝里，旁边就是悬崖。你注意看它的树冠，像不像一个人伸出手在说'来来来，进来坐'？所以叫迎客松。它不挑地方——没土，它就在石头缝里长；没水，它就喝雾气。这种倔脾气，倒是很像安徽人的性格。"

**场景三：无障碍旅游**
听障游客参观博物馆→讲解内容全部以AR字幕显示在视野下方，关键信息用手语动画辅助，震动提示参观节点。

## 四、技术方案

### 4.1 技术架构

> 核心思路：**AI能力是主角，Rokid眼镜是AI的载体**
> 我们不做底层AR开发，调用Rokid现成SDK能力，把全部精力集中在AI链路上

```
┌─────────────────────────────────────────────────────────┐
│                  Rokid AI眼镜（载体层）                   │
│  · CX-L SDK 调用摄像头画面                               │
│  · 语音模块：语音识别 + 语音播报                         │
│  · 显示模块：AR信息叠加                                  │
│  （均为SDK API调用，无需底层AR开发）                     │
└──────────────────────┬──────────────────────────────────┘
                       │
┌──────────────────────▼──────────────────────────────────┐
│                   AI 核心链路                             │
├─────────────────────────────────────────────────────────┤
│                                                          │
│  ① 视觉识别层                                           │
│  · 景点图像识别（视觉特征匹配）                          │
│  · GPS辅助粗定位                                         │
│  · 视觉+GPS融合定位                                     │
│                                                          │
│  ② 大模型理解层（Gemini）                               │
│  · 多模态输入：图像 + GPS + 用户语音                    │
│  · 景点识别 → 知识库匹配                                │
│  · 分层讲解内容生成                                     │
│  · 多语言实时翻译                                       │
│  · 开放式问答                                           │
│                                                          │
│  ③ 知识层                                               │
│  · 景点知识图谱（结构化数据）                            │
│  · 多语言讲解模板库                                     │
│  · 用户画像                                             │
│                                                          │
│  ④ 交互层                                               │
│  · ASR（语音识别）→ 语音转文本                          │
│  · TTS（文本转语音）→ 语音播报                          │
│  · 多轮对话管理                                         │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

### 4.2 技术栈

| 模块 | 技术选型 | 说明 |
|------|---------|------|
| AI核心 | Gemini API | 多模态理解 + 内容生成 + 多语言 |
| 视觉识别 | CLIP / ResNet | 景点图像特征匹配 |
| 后端服务 | Python FastAPI | 轻量接口 |
| 数据存储 | PostgreSQL + Neo4j | 景点数据 + 知识图谱 |
| 眼镜端 | Rokid CX-L SDK | 摄像头/语音/显示调用 |
| 版本管理 | Git | 协作开发 |

### 4.3 技术可行性

| 能力 | 成熟度 | 难度 | 说明 |
|------|--------|------|------|
| 视觉场景识别 | ★★★★★ | 低 | 开源模型+成熟方案 |
| 大模型文本生成 | ★★★★★ | 低 | Gemini API直接调用 |
| 语音交互（ASR+TTS）| ★★★★★ | 低 | 成熟API |
| 多语言翻译 | ★★★★★ | 低 | Gemini原生能力 |
| Rokid SDK调用 | ★★★★☆ | 中 | 需要学习，但有官方文档 |
| 知识图谱构建 | ★★★★☆ | 中 | 数据整理工作量较大 |

### 4.4 我们的差异化技术

**不是做AR，是让AI更"懂"景区**

- **位置感知 + 景区自动识别**：GPS定位自动判断当前所在景区，无感切换知识库
- **知识库自生长**：基于游客游览轨迹+停留热点，自动发现未收录的兴趣点，持续更新知识库
- **视觉定位辅助**：用图像特征匹配辅助GPS，在古建筑遮挡区域提升定位准确率
- **景点知识图谱**：基于GIS空间关系构建景点关联网络，讲解能自动串联
- **空间语义理解**：知道游客"面朝什么方向""在看哪个建筑的哪个部分"

## 五、讲解词生成引擎（核心竞争力）

> 市面上的AI讲解是把百度百科念给你听。
> 我们让AI像真人导游一样给你讲故事。
> **技术上不难，难的是Prompt调得好不好。我们已经想清楚了怎么调。**

### 5.1 为什么现有AI讲解很死板？

| 问题 | 表现 | 原因 |
|------|------|------|
| 百科式播报 | "建于1420年，高35米" | 直接拼接数据库字段 |
| 没有节奏 | 一口气讲完 | 没有分层结构 |
| 缺乏人情味 | 像机器念稿 | 没有角色设定 |
| 千人一面 | 所有人听一样的 | 没有个性化 |

### 5.2 解决方案：Prompt Engineering

**角色设定**：
> 你是一位拥有20年经验的金牌导游，专讲历史文化。
> 像在跟朋友聊天，喜欢用类比把复杂的事说简单。
> 会穿插野史八卦冷知识，偶尔自嘲开玩笑。
> 关键信息用口语表达，避免书面语。
> 会引导游客观察细节，会提问互动。

**分层讲解模板**：

| 层级 | 时长 | 触发 | 写法要求 |
|------|------|------|---------|
| 基础层 | 10秒 | 看一眼 | 一句话悬念钩子，勾兴趣 |
| 标准层 | 30秒 | 停3-10秒 | 完整小故事，起承转合 |
| 深度层 | 2分钟+ | 停10秒+ | 冷知识+情感升华+转场引导 |

**示例对比**：

| 级别 | ❌ 现有产品 | ✅ 我们的方案 |
|------|-----------|-------------|
| 基础层 | "太和殿，建于永乐十八年" | "这个大家伙，六百多岁了，比我们所有人加起来都大——停一下？" |
| 标准层 | "太和殿面阔十一间，进深五间" | "你看这台阶上的浮雕，磨得快看不清了——不是风吹的，是三百年来无数官员跪着磕头磨出来的" |
| 深度层 | 念三分钟百科 | 从一个细节切入讲历史故事，穿插冷知识，结尾引导到下一个景点 |

### 5.3 互动问答

- 游客提问时，先肯定："好问题！""你这个角度很刁钻"
- 回答像聊天不像百科
- 不知道的诚实说"这个我得查查"，不编造
- 回答完可以反问或引导延伸

### 5.4 个性化适配

首次使用快速选择（3个问题）：
1. 风格：故事版 / 专业版 / 轻松版
2. 兴趣：历史故事 / 建筑艺术 / 自然风光
3. 深度：快速浏览 / 正常 / 深度了解

### 5.5 位置感知 + 知识库自生长

> 知识库不是一次写死的，而是随着使用不断"长大"的活系统

**景区自动识别：**
- GPS定位判断当前所在景区，自动加载对应知识库
- 进入景区范围 → 自动启动讲解模式
- 离开景区 → 自动切换到通用导航模式

**知识库自动生成流程：**
```
初次进入景区
    │
    ├── 调用公开数据源（地图POI、官方介绍、公开文献）
    │         ↓
    ├── AI自动生成基础讲解框架
    │         ↓
    ├── 游客A游览，停留某点30秒以上 → 标记为"潜在兴趣点"
    │         ↓
    ├── 游客B/C/D也在此停留 → 确认为"新景点"
    │         ↓
    ├── AI基于位置+图像，自动抓取相关信息生成讲解初稿
    │         ↓
    └── 人工审核确认 → 正式入库
```

**数据来源：**
- 公开POI数据（高德/百度地图景点信息）
- 景区官方资料（官网、公众号、百科）
- 游客UGC内容（点评、游记、社交媒体打卡）
- 视觉采集（游客眼镜拍摄的画面 → AI提取特征 → 自动建档）

**持续更新机制：**
- 每次游客游览后，上传匿名轨迹+停留数据
- AI分析发现：新增热点 → 自动生成新景点讲解初稿
- 定期更新：季节性变化（花开了/雪景了）、新增展品、修缮信息
- 人工审核：生成的讲解经人工确认后正式入库，保证质量
